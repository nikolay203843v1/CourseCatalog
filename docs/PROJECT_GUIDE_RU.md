# Как понимать и объяснять проект

Этот файл лучше читать вместе с кодом в указанном порядке. Цель — не выучить
названия классов, а увидеть путь одного действия от интерфейса до базы данных.

## 1. С чего начать

1. `settings.gradle.kts` — какие модули входят в проект.
2. `CourseCatalogApplication.kt` — какие Koin-модули подключаются.
3. `app_nav_graph.xml` — первый экран и переход после входа.
4. `LoginFragment.kt` и `LoginViewModel.kt` — простой пример MVVM.
5. `Course.kt` и `CourseRepository.kt` — центр предметной области.
6. `CourseDataModule.kt` — создание Room, Retrofit и репозитория.
7. `DefaultCourseRepository.kt` — связь сети и локальной базы.
8. `HomeViewModel.kt` и `HomeFragment.kt` — полный реактивный экран.
9. `CourseListAdapter.kt` — отображение списка через AdapterDelegates.

## 2. Что происходит при запуске

Android создаёт `CourseCatalogApplication`. В `onCreate()` запускается Koin и
регистрирует зависимости. Затем `MainActivity` показывает `NavHostFragment`,
начальным destination которого является `LoginFragment`.

Пользователь вводит email и пароль. `doAfterTextChanged` передаёт строки во
ViewModel, а `combine` пересчитывает `canLogIn`. Фрагмент подписан на
`uiState` и включает кнопку только при валидных данных.

После входа открывается `MainContainerFragment`. Внутри него есть ещё один
`NavHostFragment`, связанный с `BottomNavigationView`.

## 3. Что происходит на главной

Koin создаёт `HomeViewModel` и передаёт use cases. В `init` вызывается
`refresh()`:

1. `RefreshCoursesUseCase` обращается к интерфейсу `CourseRepository`;
2. Koin подставляет `DefaultCourseRepository`;
3. Retrofit загружает JSON;
4. DTO преобразуются в `CourseEntity`;
5. Room сохраняет сущности транзакционно;
6. DAO публикует новый список через `Flow`;
7. репозиторий преобразует Entity в доменную модель `Course`;
8. ViewModel создаёт `CourseListUiState`;
9. Fragment передаёт курсы адаптеру;
10. `DiffUtil` обновляет только изменившиеся элементы RecyclerView.

## 4. Зачем нужны три модели

`CourseDto` соответствует внешнему JSON. Если сервер изменится, правки
останутся в data-слое.

`CourseEntity` соответствует таблице Room. Здесь есть служебное поле
`position`, которого нет в бизнес-модели.

`Course` — модель приложения. Она использует `LocalDate`, не содержит
аннотаций Retrofit или Room и не зависит от Android.

Разделение не является дублированием ради дублирования: у сети, базы и
бизнес-логики разные причины для изменения.

## 5. Зачем use cases

Use case имеет одно понятное назначение: наблюдать, обновить, отсортировать или
переключить избранное. ViewModel не знает методов DAO и не формирует HTTP
запросы.

Для небольшого задания часть use cases выглядит тонкой обёрткой. Их польза
проявляется при росте правил, повторном использовании логики на нескольких
экранах и независимом тестировании.

## 6. Основные конструкции Kotlin и Android

- `suspend` — функция может приостанавливаться без блокировки потока.
- `Flow<T>` — асинхронная последовательность значений.
- `StateFlow<T>` — Flow, который всегда хранит текущее состояние.
- `combine` — пересчитывает результат при изменении любого входного Flow.
- `viewModelScope` — отменяет корутины при окончательном удалении ViewModel.
- `repeatOnLifecycle` — запускает сбор Flow только в нужном lifecycle-состоянии.
- `Result` и `runCatching` — передают успех или ошибку без падения приложения.
- `internal` — класс доступен внутри своего Gradle-модуля.
- `by viewModel()` — Koin создаёт ViewModel и передаёт её зависимости.
- ViewBinding — типобезопасный доступ к View из XML без `findViewById`.

## 7. Вопросы, которые может задать техлид

### Почему UI читает Room, а не результат Retrofit напрямую?

Так база становится единым источником истины. Сетевое обновление меняет базу,
Room публикует Flow, и одинаковый список получают главная и избранное.

### Почему domain не зависит от data?

Бизнес-правила должны знать интерфейс репозитория, но не способ хранения.
Направление зависимости инвертировано: data реализует контракт domain.

### Почему Koin, а не Dagger/Hilt?

В задании разрешены Dagger или Koin. Для небольшого проекта Koin даёт короткую
и читаемую конфигурацию. Компромисс — часть ошибок графа зависимостей
обнаруживается во время запуска, а не компиляции.

### Почему Flow, а не LiveData?

Flow удобно объединять, преобразовывать и использовать во всех слоях, включая
чистый Kotlin domain. Во Fragment сбор привязан к lifecycle.

### Почему избранное не сбрасывается после обновления?

Перед маппингом ответа сохраняется карта локальных значений по id. Для уже
известного курса она имеет приоритет над `hasLike` из JSON.

### Почему AdapterDelegates, если тип элемента один?

Это требование задания. Даже с одним типом delegate изолирует binding карточки,
а при добавлении баннера, skeleton или секции список расширяется без большого
`when` внутри адаптера.

### Что бы ты изменил в production?

Добавил бы MockWebServer и instrumented-тест DAO, retry, логирование только для
debug, CI, аналитику ошибок и серверную авторизацию. Для больших списков
рассмотрел бы Paging 3.

## 8. Как учить проект

Не пытайся сразу запомнить библиотеки. Выбери сценарий и пройди его от нажатия
до результата. Например:

```text
bookmarkButton
  → HomeViewModel.toggleFavorite
  → ToggleFavoriteUseCase
  → CourseRepository
  → DefaultCourseRepository
  → CourseDao.toggleFavorite
  → Room Flow
  → новый uiState
  → обновление карточки
```

После этого временно измени поведение сам: порядок сортировки, текст пустого
состояния или правило email. Если можешь предсказать результат, внести правку,
проверить её и объяснить путь данных — ты уже не просто пересказываешь код.
